# Notion API Weekly Report

Use this skill to summarize recent agent work into database rows in a Notion weekly report. Prefer a small set of system-oriented weekly work items over a detailed transcript of conversations, tool calls, or learning steps.

## Workflow

1. Inspect recent Codex threads with `list_threads` and `read_thread` when the user asks to summarize left-side or recent Codex conversations.
2. Extract work into rows:
   - `任务`: final weekly work item. Write the actual work content, not the intermediate action used to discover or implement it. Also support databases that call this `任务名称`.
   - `状态`: use existing status names such as `已完成`, `进行中`, `未开始`, or `下周目标`.
   - `负责人`: set through `--owner` or local `NOTION_WEEKLY_OWNER` when available.
   - `系统`: use as system/project category. Also support databases that call this `任务类型`.
   - `备注`: 备注默认留空。Fill it only for important numbers, verification results, risks, or user-requested detail. Also support databases that call this `描述`.
   - `创建时间` / date properties: use the real upload time generated by `scripts/upload_rows.mjs`.
3. Group candidate work before writing rows:
   - 默认一周控制在 5-8 条左右，除非用户明确要求完整明细。
   - Merge multiple conversations, debugging steps, documentation lookups, script edits, and configuration changes when they serve the same weekly work theme.
   - Do not create one row per chat window, one row per tool call, or one row per learning step.
   - Prefer task names like `完善Notion周报自动上传skill及本地配置流程` over process names like `获取并梳理 Codex Skills 用法`.
4. Filter candidate work before writing rows:
   - Include only concrete system/project/coursework/tooling work that maps to an existing `系统` / `任务类型` option.
   - Prefer rows with an action, result, decision, bug diagnosis, configuration change, upload/write action, verification number, or next system-related task.
   - Exclude casual chat, social conversation, general explanations without a deliverable, repeated clarification, meta discussion about the assistant, and one-off questions that did not produce project/system output.
   - Exclude standalone learning-only threads unless they directly support a system work item; if they do, merge them into that system item.
   - When unsure whether a conversation is report-worthy, keep it out or ask the user; do not pad the weekly report with low-signal rows.
5. Classify `系统` by the system actually being served, using only options already present in the database:
   - Always inspect the database options first with `--dry-run` or a database fetch.
   - Prefer the user's existing project/system abbreviations. Known options often include `SMDB`, `DBserv`, `HSAS`, `CDAE`, `IQTS`, and `ETMS`.
   - 分类主要看这件事实际是在服务哪个系统，不是看用了哪个工具、查了什么文档、在哪个 agent 里做。
   - Use `DBserv` for database coursework, QQ/Excel grade work, QQEX, Tencent Docs, grade entry, or database-system teaching tasks when available.
   - Use `IQTS` for IQTS development, reconstruction, local environment setup, framework setup, module/API configuration, and AI workflow work that serves IQTS.
   - Use `ETMS` for ETMS bug fixing and feature issue handling, including Web/backend/import/activity/announcement/student-related problems, plus tooling work that mainly serves ETMS.
   - Do not classify by the tool used, the documentation source, or the agent window. Classify by the system that benefits from the work.
   - Merge AI learning, Context7 documentation checks, AGENTS.md/Skills/GSD research, and similar workflow work into the system they support. For example, if it supports IQTS development flow, classify it as `IQTS`.
   - If a tooling item is genuinely cross-system, classify it by the current user context or the primary beneficiary. If still unclear, show a draft and ask before uploading.
   - If no obvious existing option fits, stop and ask the user to choose one instead of creating a new `系统` option.
6. Confirm Notion API access:
   - Token is valid if `GET /v1/users/me` succeeds.
   - Target database is visible if `POST /v1/search` returns it or `GET /v1/databases/{database_id}` succeeds.
   - If API returns 404, tell the user to add the integration from Notion integration management or page/database access settings.
7. Discover database schema before upload:
   - Use `scripts/upload_rows.mjs --dry-run`, search, or fetch the database.
   - Map only existing properties and existing select/multi-select/status options.
   - Do not invent properties unless the user asks to modify the database.
8. Upload with `scripts/upload_rows.mjs`.
9. Return the database row count and representative URL.

## Fast Weekly Upload Path

When `NOTION_TOKEN` and `NOTION_WEEKLY_DATABASE` are already fixed for the team:

1. Create the weekly rows JSON after grouping by system and weekly work theme.
2. Run one `--dry-run` with the normal script to verify the database, owner, status options, and system options.
3. If the dry-run passes, run the same script without `--dry-run`.

Do not switch tools or create one-off upload scripts unless the normal script is genuinely broken after retrying. The script already retries transient Notion/network failures.

## Row Style

- Keep rows short and scan-friendly.
- Split unrelated system work into separate rows, but merge small steps that belong to the same work theme.
- Do not summarize every conversation. Weekly rows are for meaningful work items, not a chat transcript.
- Leave `备注` empty by default; include measurable verification only when it materially helps the weekly report.
- Put future work as `未开始`, `进行中`, or `下周目标` according to the database's existing status options.

Example rows:

- `基于AI工作流继续推进IQTS重构前的准备与配置` / `已完成` / `IQTS`
- `修复IQTS本地开发环境中IDEA Docker运行配置问题` / `已完成` / `IQTS`
- `排查HeyGen插件连接失败和OAuth回调异常` / `已完成` / `ETMS`
- `完善Notion周报自动上传skill及本地配置流程` / `已完成` / `ETMS`

## Notion Requirements

- The integration needs at least `Read content`, `Update content`, and `Insert content`.
- The integration must be granted access to the target database or its parent page. Having a token alone is not enough.
- Use Notion API version `2022-06-28` unless the user asks for another version.

## Script Usage

The script expects JSON input so rows can vary by week.

```powershell
$env:NOTION_TOKEN = "<token>"
node path\to\scripts\upload_rows.mjs --database "<database-id-or-url>" --input ".\weekly-rows.json"
Remove-Item Env:\NOTION_TOKEN
```

If `NOTION_WEEKLY_DATABASE` is set, omit `--database`:

```powershell
node path\to\scripts\upload_rows.mjs --input ".\weekly-rows.json"
```

To set a responsible person for one run:

```powershell
node path\to\scripts\upload_rows.mjs --input ".\weekly-rows.json" --owner "<notion-user-id-or-visible-name-or-email>"
```

To set a computer-local default responsible person without changing the skill for other members:

```powershell
[Environment]::SetEnvironmentVariable("NOTION_WEEKLY_OWNER", "<notion-user-id>", "User")
```

The script uses `--owner` first, then `NOTION_WEEKLY_OWNER`. If neither is set, it leaves the responsible person empty.

Input JSON shape:

```json
{
  "items": [
    {
      "name": "完成 4月16日数据库系统作业批改并写入成绩表 S 列",
      "status": "已完成",
      "system": "DBserv",
      "description": "完成 43 份提交批改，保留 3 个空白，核对合计 415。"
    }
  ]
}
```

## Persistent Setup

For weekly use on Windows, store the token, stable database, and local owner once:

```powershell
[Environment]::SetEnvironmentVariable("NOTION_TOKEN", "<token>", "User")
[Environment]::SetEnvironmentVariable("NOTION_WEEKLY_DATABASE", "<database-id-or-url>", "User")
[Environment]::SetEnvironmentVariable("NOTION_WEEKLY_OWNER", "<notion-user-id>", "User")
```

Open a new PowerShell/Codex session after setting them. To replace any value, run the same command with the new value.

## Owner ID Notes

Notion People properties require a Notion user id. Names and emails only work when the integration can see them through `/v1/users`. If `/users` does not return the desired person, ask the user to manually select themselves in any row's `负责人` property, then read that row through the API and copy `properties.负责人.people[0].id` into `NOTION_WEEKLY_OWNER`.

## Safety

- Never store the user's Notion token in the skill, repo, generated report, logs, or final answer.
- Prefer reading the token from `NOTION_TOKEN`.
- If multiple people give tokens, switch `NOTION_TOKEN` locally before upload and verify the target database with `--dry-run`.
- Avoid printing request headers or full token-bearing commands in final answers.
- Do not create new `系统` or `任务类型` options accidentally. If an item uses a missing system option, stop and ask the user to choose one existing option.
